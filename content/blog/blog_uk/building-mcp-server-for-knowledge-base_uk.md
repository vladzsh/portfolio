## Проблема

Наша внутрішня база знань була хаосом. Архітектурні рішення жили у Confluence. Специфікації проектів — у трекері задач. Нотатки зустрічей плавали в Google Docs. Гайди з онбордингу існували як випадкові markdown-файли.

Понад 500 документів, розкиданих по п'яти джерелах даних, доступних для пошуку тільки якщо знаєш, який інструмент відкрити і що шукати.

Потім ми побудували MCP-сервер, і Claude навчився шукати по всьому одразу.

## Що таке MCP

**Model Context Protocol** — це відкритий стандарт, який дозволяє AI-моделям викликати зовнішні інструменти. Замість ручного вставляння контексту в промпт, ви відкриваєте інструменти — такі як `semantic_search` чи `get_document` — і модель викликає їх, коли потребує інформацію.

Уявіть це як API, яким AI-моделі вміють користуватися.

## Архітектура

Наша система має три шари:

**Шар 1: Django-бекенд** — інгестить документи з кількох джерел, розбиває на чанки, генерує ембедінги, зберігає у Qdrant (векторна база даних) і відкриває REST API для пошуку.

**Шар 2: MCP-сервер** — легкий FastMCP HTTP-сервіс, який транслює MCP tool-виклики у запити до Django API. Обробляє автентифікацію і форматує відповіді для моделі.

**Шар 3: Claude** — підключається до MCP-сервера і викликає інструменти за потреби під час розмов.

Сам MCP-сервер крихітний — приблизно 100 рядків. Вся складність живе у Django-бекенді.

## MCP-сервер

Ми використали **FastMCP** у stateless HTTP-режимі. Сервер відкриває чотири інструменти:

```python
mcp = FastMCP(
    "knowledge-base",
    stateless_http=True,
    host="0.0.0.0",
    port=8032,
)

@mcp.tool(description="Пошук по базі знань за семантичною подібністю")
async def semantic_search(
    ctx: Context,
    query: str,
    top_k: int = 5,
    tags: list[str] | None = None,
    categories: list[str] | None = None,
) -> str:
    token = _extract_token(ctx)
    result = await kb_client.search(token, query=query, top_k=top_k, tags=tags, categories=categories)
    return json.dumps(result, indent=2)
```

Автентифікація проходить наскрізно від Claude через MCP-сервер до Django-бекенду — все через Bearer-токени. MCP-сервер ніколи не зберігає креденшли; він витягує токен з контексту запиту і передає далі.

Інші три інструменти — `get_document`, `get_document_chunks` і `list_filters` — слідують тому ж патерну: витягти токен, проксувати до Django, повернути JSON.

## Пайплайн інгестії

Документи приходять з п'яти джерел: сторінки Confluence, статті трекера задач, Google Sheets, Google Docs і ручні завантаження. Кожне джерело має свій лоадер, але всі подають у один пайплайн:

**Отримати** → **Конвертувати в Markdown** → **Розбити на чанки** → **Зробити ембедінг** → **Зберегти в Qdrant**

### Стратегія чанкінгу

Ми використовуємо підхід з двома сплітерами. Спочатку пробуємо розбити по Markdown-заголовках:

```python
splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=[("#", "H1"), ("##", "H2"), ("###", "H3")]
)
```

Якщо Markdown-розбиття дає погані результати — один гігантський чанк або чанки, що перевищують подвійний цільовий розмір — ми переходимо на рекурсивне розбиття по символах:

```python
RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=100,
    separators=["\n\n", "\n", ". ", "! ", "? ", " ", ""]
)
```

Підхід Markdown-first зберігає структуру документа. Секція під назвою "## Потік автентифікації" залишається одним чанком, що означає — результати пошуку повертають зв'язний контекст, а не довільні 500-символьні зрізи.

### Асинхронні ембедінги через Celery

Ембедінг — найповільніша частина інгестії, кожен чанк потребує API-виклику до моделі OpenAI `text-embedding-3-small`. Ми обробляємо це асинхронно через Celery-задачі, що тригеряться Django-сигналами:

```python
@receiver(post_save, sender=Chunk)
def handle_chunk_status_change(sender, instance, **kwargs):
    if instance.status == Status.APPROVED:
        add_chunk_to_vector_store_task.delay(instance.id)
    elif was_previously_approved(instance):
        delete_embedding_from_qdrant(point_id=instance.id)
```

Коли чанк затверджується, Celery-задача генерує його ембедінг і зберігає в Qdrant з метаданими (назва, URL джерела, теги, категорії, проекти). Якщо чанк пізніше модифікується або видаляється, сигнал видаляє його з векторного сховища.

Задача включає auto-retry з експоненціальним backoff — rate limits OpenAI реальні, і краще повторити, ніж втратити документи.

## Пайплайн пошуку

Коли Claude викликає `semantic_search("як працює наш OAuth2-потік")`, ось що відбувається:

**1. Ембедінг запиту.** Текст запиту конвертується у 1536-вимірний вектор тією ж моделлю ембедінгу, що й документи.

**2. Пошук у Qdrant.** Запускається пошук за косинусною подібністю з порогом 0.3 і опціональними фільтрами метаданих:

```python
search_result = client.query_points(
    collection_name="documents",
    query=embedding,
    limit=top_k,
    query_filter=build_filter(tags, categories, projects),
    score_threshold=0.3,
)
```

Фільтр підтримує OR-логіку — пошук з `tags=["auth", "security"]` повертає документи з тегом *будь-яким* з них.

**3. Завантаження контенту чанків.** Qdrant зберігає ембедінги та метадані, але повний текст живе у PostgreSQL. Ми отримуємо фактичний контент чанків з бази Django за point ID.

**4. Повернення структурованих результатів.** Кожен результат включає контент чанка, назву документа, URL джерела, оцінку подібності та метадані. Claude використовує це для відповіді на запитання користувача з атрибуцією.

## Що ми дізналися

**Розмір чанка важливіший, ніж здається.** Ми починали з 1000 символів і отримували розмиті результати. Зменшення до 500 з overlap 100 дало набагато точніші збіги — кожен чанк відповідає приблизно одному концепту або одній секції.

**Markdown-розбиття краще за наївне.** Розбиття по заголовках зберігає зміст. 500-символьне вікно, яке починається посеред речення і закінчується посеред абзацу — це шум. Markdown-секція, яка виявилася на 600 символів — це сигнал.

**MCP-сервер має бути тупим.** Перший дизайн розміщував логіку пошуку в MCP-сервері. Погана ідея — це означало деплоїти два сервіси щоразу, коли ми змінювали алгоритм пошуку. Перенесення всього інтелекту у Django-бекенд і збереження MCP-сервера як тонкого проксі зробило все простішим.

**Bearer token passthrough — найпростіша модель автентифікації.** MCP-сервер не управляє користувачами чи сесіями. Він просто прокидує токен. Django валідує його, перевіряє дозволи і повертає результати. Одна система автентифікації, а не дві.

## Другий MCP-сервер

Патерн нам настільки сподобався, що ми побудували другий MCP-сервер — цього разу для нашого трекера задач. Той самий підхід: FastMCP HTTP-сервіс, прокидування токена, доменно-специфічні інструменти (пошук задач, деталі задач, управління спринтами, трекінг часу).

Різниця: цей використовує **OAuth2 з PKCE** для автентифікації замість простих Bearer-токенів, бо трекер задач цього вимагає. `OAuthProxy` FastMCP обробляє потік — URL авторизації, callback, обмін токенів, оновлення — тому MCP-сервер залишається тонким.

Між двома серверами Claude тепер може шукати по нашій документації і нашому інструменті управління проектами в одній розмові. "Що каже архітектурний документ про автентифікацію?" і потім "Покажи відкриті тікети, пов'язані з цим" — без перемикання вкладок.

## Чи варто будувати?

Якщо у вашої команди є внутрішня документація, яка реально потрібна людям, але яку вони не можуть знайти — так. MCP-сервер — це легка частина. Складна частина — пайплайн інгестії та стратегія чанкінгу. Зробіть їх правильно, і все інше випливає.

Почніть з одного джерела даних, доведіть якість пошуку до потрібного рівня, потім додавайте більше джерел. Ми почали з Confluence, довели працездатність, потім додали трекер задач, Google Docs і Sheets протягом наступних тижнів.
